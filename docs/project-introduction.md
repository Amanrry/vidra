# Vidra 项目介绍与商业/技术构想

## 1. 项目概述

Vidra 是一个面向开放视频场景的多模态视频理解与问答系统。用户上传一段视频后，系统会自动完成视频抽帧、目标检测、目标裁剪、帧图像描述、音频转录、多模态向量索引和大模型问答。最终用户可以在 Web 界面中查看 YOLO 检测可视化结果，也可以像使用 ChatGPT 一样用自然语言询问视频内容。

项目的核心目标不是只做单一领域分类器，而是构建一个可以处理任意类型视频的通用视频智能分析底座。系统不依赖人工维护的标签配置文件，也不要求用户预先定义业务领域。它通过视觉检测、图像描述、语音文本、向量检索和大语言模型协同工作，把原始视频转化为可检索、可解释、可问答的结构化知识。

## 2. 项目现有功能

### 2.1 视频上传与异步处理

Vidra 支持通过 Gradio Web 界面上传视频。上传完成后，前端会立即返回，不会等待重模型处理全部结束。后端使用进程内优先级队列处理视频任务，支持任务排队、状态轮询、重复任务拦截和优雅关闭。

用户界面会显示 queued、processing、partial、complete、failed 等状态，并在处理进度变化时刷新结果区域。这样可以避免长时间阻塞页面，也符合真实应用中“上传后后台处理、前台持续查看进度”的产品体验。

### 2.2 视频帧采样

系统使用可配置的时间间隔对视频进行抽帧，例如每 2 秒采样一帧，并限制最大抽帧数量，避免长视频造成处理成本失控。每个采样帧都会保存：

- 视频 ID
- 帧 ID
- 帧序号
- 时间戳
- 图片路径

这些元数据后续会进入 SQLite 和向量数据库，成为视频检索和问答的基础证据。

### 2.3 YOLO 目标检测与目标裁剪

Vidra 使用 YOLO 模型对采样帧进行目标检测。每个检测结果包含目标类别、置信度、边界框坐标、所属帧、时间戳等信息。

系统还会基于边界框裁剪目标图片，并保存：

- 带检测框的可视化帧
- 单个目标 crop 图片
- 检测对象上下文记录
- crop 上下文记录

这些结果既可以在前端直接展示，也可以进入向量索引，用于后续图文检索和问答。

### 2.4 图像描述与音频转录

对每个采样帧，系统使用 BLIP 图像描述模型生成自然语言 caption，例如“a person standing near shelves”。这使视频帧不只停留在像素层和目标标签层，而是具备可被语言模型理解的语义描述。

如果视频包含音频，系统会使用 Whisper 进行音频转录，生成带时间戳的 transcript 片段。如果视频没有音频或音频处理失败，系统不会直接终止整个任务，而是保留已有视觉结果，将状态标记为 partial 或带 warning 的完成状态。

### 2.5 多模态向量索引与检索

系统使用 SigLIP 提取文本和图像特征，将 caption、transcript、object label、crop、frame 等上下文统一写入本地 ChromaDB 向量数据库。

用户可以输入文本查询，例如“person walking”“bowl on shelf”“someone riding a bicycle”，系统会将问题编码为向量，在本地向量库中召回最相近的视频证据。该能力支撑两类功能：

- Search：直接搜索视频中的相关片段、目标和图像。
- QA：把召回证据交给大语言模型生成回答。

### 2.6 RAG 问答

Vidra 的问答模块采用 RAG 思路。用户提出问题后，系统会：

1. 将问题转成文本向量。
2. 从 ChromaDB 中召回相关 caption、transcript、object、crop、frame 证据。
3. 使用 ContextBuilder 组合上下文。
4. 在 prompt 中加入防幻觉约束，要求模型只根据视频证据回答。
5. 调用 OpenAI-compatible LLM 服务生成最终回答。

系统只支持 OpenAI-compatible server，因此可以接入本地或远程兼容接口，例如本地大模型服务、云端兼容 API 或其他教学环境提供的模型服务。

### 2.7 Gradio 交互界面

前端界面被设计成左右布局：

- 左侧展示视频处理结果，包括源视频、YOLO 可视化图、检测表格、证据表格和搜索结果。
- 右侧是 AI Chat 对话框，用户可以直接围绕当前视频提问。

这种布局符合真实产品中“左侧看证据、右侧问问题”的工作方式，便于用户一边查看视频分析结果，一边通过自然语言探索视频内容。

### 2.8 CLI 与测试入口

除了 Web 界面，Vidra 还提供命令行能力：

- `process`：同步处理视频，适合测试和演示。
- `enqueue`：提交异步处理任务。
- `worker`：消费队列任务。
- `status`：查看处理状态。
- `search`：搜索视频上下文。
- `ask`：对指定视频发起问答。

命令行能力便于后续扩展成批处理、后台服务或自动评测流程。

## 3. 系统实现架构

### 3.1 分层架构

Vidra 采用较清晰的后端分层结构：

- `models`：定义 VideoRecord、FrameRecord、DetectionRecord、ProcessingJob、QAResponse 等数据契约。
- `tools`：封装具体 AI 工具，包括抽帧、图像描述、音频转录、YOLO 检测。
- `services`：实现应用服务、视频处理编排、队列、进度、向量索引、上下文构建和问答。
- `storage`：管理 SQLite schema、数据库事务和运行目录。
- `ui`：提供 Gradio 事件处理函数。
- `app.py`：组合 Gradio 应用。
- `cli.py`：提供命令行入口。

这样的设计让模型调用、业务编排、数据库写入和 UI 展示相互隔离，后续替换模型或扩展前端时不会影响整个系统。

### 3.2 核心处理 pipeline

完整 pipeline 如下：

```text
视频上传
  -> 创建 VideoRecord 与运行目录
  -> 进入处理队列
  -> 抽帧
  -> 帧图像 caption
  -> 音频转录
  -> YOLO 检测与 crop 生成
  -> 上下文持久化
  -> SigLIP 向量化
  -> ChromaDB 索引
  -> 生成 report.json / detections.csv / summary.md
  -> 前端展示与 RAG 问答
```

### 3.3 数据持久化设计

项目使用 SQLite 保存核心结构化数据，包括：

- `videos`：视频基本信息和处理状态。
- `video_context`：帧、caption、transcript、object、crop、metadata 等统一上下文。
- `processing_jobs`：队列任务、阶段、进度、错误和完成时间。
- `lineage`：记录每个工具和模型的处理来源，便于追踪。
- `processing_idempotency`：保存阶段幂等信息，避免重复写入。

运行产物按视频 ID 存放在 `data/runs/<video_id>/` 下，包括源视频、帧图片、YOLO 可视化图、crop 图片和报告文件。

### 3.4 幂等、事务与可追踪性

处理服务层提供批量写入、事务和幂等 key 机制。每个阶段写入数据时会携带：

- video_id
- tool_name
- model_name
- run_id
- idempotency_key
- parameters

这样做可以保证同一个视频重复处理时不会无限插入重复上下文，也能在出错后更容易定位是哪个模型、哪个阶段、哪批数据导致问题。

### 3.5 异步队列与进度管理

系统使用本地进程内处理队列。队列支持：

- 优先级任务。
- 同一视频重复任务拦截。
- 后台 worker。
- 任务状态查询。
- 处理进度持久化。

这种方案适合课程项目和轻量级产品原型。相比直接在请求线程中处理视频，它更稳定，也更接近真实业务系统。

### 3.6 上下文构建与问答

ContextBuilder 是问答质量的关键。它会综合多种证据：

- caption 证据
- transcript 证据
- object label 证据
- crop/frame 图像证据
- timestamp window 证据
- keyword search 和 semantic search 结果

最终 prompt 会强调：如果证据不足，应明确说明不能确定，而不是编造答案。

## 4. 根据现有条件设计一个商业项目

### 4.1 商业项目名称

可以将 Vidra 包装为一个商业项目：智能视频内容审计与问答平台。

该平台面向企业、学校、实验室、零售门店、交通运营方等拥有大量视频资料的组织，帮助他们把视频从“只能人工观看的资料”转化为“可以搜索、可以统计、可以问答的知识资产”。

### 4.2 目标客户

目标客户可以分为几类：

- 校园与公共机构：分析校园监控、活动视频、安全巡检视频。
- 零售企业：分析货架、客流、商品陈列和异常行为。
- 实验室与工厂：分析设备操作、实验过程、安全规范执行情况。
- 交通与安防部门：分析车辆、人群、道路事件和异常场景。
- 内容管理团队：快速检索大量视频素材中的人物、物体和事件。

### 4.3 核心业务场景

1. 视频内容搜索  
用户上传大量视频后，可以搜索“有人骑自行车”“货架上有杯子”“设备旁边有人操作”等自然语言问题，系统返回相关片段和证据图。

2. 智能视频问答  
用户不需要手动拖动进度条查看视频，可以直接询问“视频中出现了哪些物体？”“什么时候有人经过货架？”“是否出现了自行车或摩托车？”。

3. 目标检测可视化  
系统自动生成带检测框的图像和目标裁剪图，便于人工复核和报告展示。

4. 视频结构化归档  
将原始视频变成 caption、transcript、object、crop、timestamp 等结构化数据，便于长期查询和数据再利用。

5. 自动报告生成  
系统可输出检测结果 CSV、summary.md 和 JSON 报告，适合课程展示、企业内部审计和业务复盘。

### 4.4 产品形态

第一阶段可以做成本地部署的单机版系统：

- 用户通过 Web 页面上传视频。
- 后台自动处理。
- 前端查看检测结果。
- 通过 Chat 问答视频内容。
- 所有数据存储在本地，适合隐私要求较高的场景。

第二阶段可以扩展为团队版：

- 多用户登录。
- 视频项目空间。
- 批量上传。
- 任务队列管理。
- 权限控制。
- 报告分享。

第三阶段可以扩展为 SaaS 或私有化部署：

- 支持对象存储。
- 支持 GPU worker 集群。
- 支持企业知识库。
- 支持视频实时流接入。
- 支持按行业定制模型。

## 5. 商业构想

### 5.1 价值主张

Vidra 的商业价值可以概括为：降低视频理解成本，让企业视频资料可搜索、可追溯、可问答。

传统视频分析依赖人工查看，效率低、遗漏率高、难以复用。Vidra 将视频拆解为结构化、多模态、可检索的数据，使用户可以用自然语言直接访问视频内容。

### 5.2 差异化优势

1. 开放场景能力  
系统不要求预先绑定某个特定行业标签，可以处理校园、交通、商超、实验室等不同类型视频。

2. 多模态证据融合  
不只依赖目标检测，还融合 caption、transcript、crop 图像和向量检索，因此问答依据更丰富。

3. 本地化部署友好  
SQLite、ChromaDB、本地模型和 OpenAI-compatible LLM 使系统适合教学、企业内网和隐私敏感环境。

4. 可解释输出  
回答不是凭空生成，而是可以追溯到时间戳、检测对象、caption、crop 和报告文件。

5. 成本可控  
通过抽帧、最大帧数限制、异步队列和本地模型，可以控制计算成本。

### 5.3 收费模式设想

可设计三种商业模式：

- 单机专业版：按授权收费，适合学校实验室、小企业和课程项目部署。
- 企业私有化版：按服务器数量、GPU 数量或视频处理量收费，提供部署和定制服务。
- SaaS 版：按视频分钟数、存储容量、问答次数或团队账号数量收费。

### 5.4 可落地的 MVP

最小商业可行版本可以包含：

- 视频上传。
- 后台处理队列。
- YOLO 检测可视化。
- 目标 crop 展示。
- caption/transcript 生成。
- 本地向量搜索。
- 视频问答。
- 报告导出。

这与当前项目已经实现的能力基本一致，后续只需要补充用户体系、批量管理、权限控制和更完整的前端体验，就可以向商业原型过渡。
